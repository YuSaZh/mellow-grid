MellowGrid is a visual Bento editor for a personal homepage. It separates the public page and the editor into two clear paths: visitors get a lightweight static page generated by Astro, while React only loads when you open `/editor`.

The goal is not to build a multi-tenant SaaS product. MellowGrid is designed as a personal, long-term-maintainable homepage workflow: visually polished, lightweight in public, intuitive to edit, and ready to publish through static output or standalone HTML export.

## Why MellowGrid

- **Static first**: the public homepage is Astro static output by default and does not require a runtime backend.
- **Editor on demand**: React, Zustand, and editor interactions only load on `/editor`.
- **Bento visual system**: soft rounded cards, modular tiles, and a compact grid for links, text, and dividers.
- **WYSIWYG-oriented**: the editor reuses the page rendering path to reduce differences between editing and publishing.
- **Personal-friendly**: local drafts, JSON backup, and static HTML export fit a one-person homepage workflow.

## Route Design

```txt
/         ultra-light static personal homepage
/editor   React editor, JavaScript loaded only while editing
```

| Route | Description |
| --- | --- |
| `/` | Public homepage rendered from `defaultPageConfig`. |
| `/editor` | Visual editor loaded with `client:only="react"`. |

## Features

- Edit profile, bio, location, and contact links.
- Add, delete, select, and edit Bento modules.
- Drag modules around the grid and resize them.
- Customize link card colors, backgrounds, gradients, and icons.
- Import and export JSON configuration.
- Save browser-local drafts.
- Export standalone static HTML.
- Keep the public homepage free from full editor dependencies.

## Active Widgets

MellowGrid currently keeps three core widgets to avoid premature complexity:

| Widget | Purpose |
| --- | --- |
| `link` | Social links, external links, and button-like cards. |
| `text` | Introductions, notes, and short content cards. |
| `divider` | Borderless section titles for splitting the right-side grid into content groups. |

## Project Structure

```txt
src/pages/index.astro          Static homepage route
src/pages/editor.astro         React editor island route
src/app/editor/                Editor UI, state, and inspectors
src/components/page/           Shared homepage and editor preview renderers
src/lib/page-config/           Page config types, defaults, and normalization
src/lib/page-export/           Static HTML export helpers
src/lib/widgets/registry.ts    Widget registry
src/widgets/                   Widget implementations
data/pages/username.json       Example import/export config
```

`dist/` and `.astro/` are generated directories and are ignored by git. Regenerate them with `npm run build` when needed.

## Editing and Publishing Workflow

1. Open `/editor`.
2. Adjust the profile, link cards, text cards, and dividers.
3. Save the draft to browser `localStorage`.
4. Export JSON for backup, or export static HTML for standalone publishing.

The current version does not include a runtime backend, page-save API, file storage adapter, or Docker deployment chain. That trade-off keeps the project focused as a personal static homepage tool rather than a long-running server application.

## Data Model

Pages are described by `PageConfig`. The key fields are:

- `profile`: left-side profile panel.
- `theme`: background, text, card, accent, radius, and shadow settings.
- `layout`: right-side Bento grid positions and sizes.
- `widgets`: right-side module instances and props.
- `updatedAt`: last update timestamp.

Related files:

- `src/lib/page-config/types.ts`
- `src/lib/page-config/defaults.ts`
- `src/lib/page-config/normalize.ts`

## Maintenance Principles

- Keep the public homepage lightweight and free from editor-only dependencies.
- Register new right-side modules through the widget registry.
- Keep the profile as an independent area instead of mixing it into the widget system.
- Keep sample data generic with placeholders such as `Username`, `Location`, and example URLs.
- Do not reintroduce a backend, Docker, or file-mode storage layer unless there is a clear requirement.
